# Roadmap: Contract IDE

## Overview

Contract IDE ships in one milestone — 13 phases, executed in dependency order, ending in a working 4-beat live demo (per `.planning/demo/presentation-script.md`) backed by a fully harvest-first intent substrate.

**Visual model (locked 2026-04-24, see `CANVAS-PURPOSE.md`):** The canvas renders flows as vertical participant chains — rendered iframe trigger for UI surfaces, structured Stripe-API-docs-style cards for backend endpoints / cron / webhooks. Atom chips overlay components on the iframe (UI mode) or sit beside the card (backend mode). Sidebar replaces L0/L1 zoom. The L0–L4 hierarchy in `.contracts/*.md` frontmatter is preserved unchanged as the data model — only the visual treatment is new. The redesign retroactively touches Phase 3 (visual treatment superseded; react-flow infra preserved), significantly reshapes Phase 9 (Babel plugin + JSX-aligned seeds + flow contracts + backend frontmatter sections), and prompts a scrap-and-replan of Phase 13's existing plans against the new node renderers.

**Phases 1–9 — contract substrate as the IDE primitive.** Phases 1–2 establish the foundation (Tauri shell, SQLite schema with indexes + FTS5 from day 1, contract file format with stable YAML frontmatter) that every later component reads from. Phase 3 builds the graph canvas on top of that data. Phase 4 adds the inspector (contract / code-with-range-highlight / preview / receipts tabs) and Monaco editor with WKWebView worker mitigation validated immediately. Phase 5 (MCP server sidecar exposing the `find_by_intent` / `get_contract` / `list_drifted_nodes` / `update_contract` tools) can run in parallel with Phase 4 after Phase 2 is complete. Phase 6 runs the LLM-backed derivation pipeline. Phase 7 is pure `notify`-watcher drift detection (shipped and verified 2026-04-24). Phase 8 is the largest — it absorbs the PostToolUse hook, the agent loop, receipts, cherrypick, AND cross-level propagation (PROP-01..04) per `.planning/research/contract-form/PROPAGATION.md`: schema migration to `format_version: 3`, canonical Rust section parser, rollup detection with amber/gray visuals reusing Phase 7's shipped watcher machinery, per-session intent journal, and pin-aware reconcile panel. Phase 9 completes the contract-substrate surface: mass semantic edit (FTS + embedding matching, section-weighted), non-coder "Copy Mode" pill, and `contract-ide-demo` seeding (Next.js + Auth + Prisma + Stripe + Mailchimp adapters) for the delete-account scenario locked in `.planning/demo/scenario-criteria.md` § Committed Scenario.

**Phases 10–13 — harvest-first intent substrate (completes the demo).** The substrate generalizes from contracts to the full intent layer (decisions, constraints, open questions, attempts) distilled automatically from every Claude Code session. Phase 10 ships the session watcher + filter pipeline (without this, no substrate gets populated). Phase 11 ships the distiller + constraint store + contract-anchored retrieval, with the `Delegate to agent` Inspector button (the entire Beat 1 → Beat 2 transition runs through this). Phase 12 ships fact-level + intent-level supersession — the moat (the Beat 3 orange-flag moment IS this claim). Phase 13 ships the substrate UI surface (Cmd+P by intent, canvas substrate-state coloring, PR-review intent-drift, mocked two-laptop `Sync` affordance) and end-to-end demo polish. **All four are demo-load-bearing** — drop any one and the scripted demo doesn't run. Post-milestone polish (real multi-machine sync, Substrate-tab Inspector, broader intent-drift surfaces) is deferred to v3, but everything required to demo runs end-to-end at milestone close. See `.planning/VISION.md` for the thesis and `.planning/ROADMAP-REVISIONS.md` for the historical scoping discussion that produced the current 13-phase plan.

### Phase 8: Agent Loop + Receipts + PostToolUse Hook + Cherrypick + Cross-Level Propagation
**Goal**: Demo Beat 1 works end-to-end (user locates a node, edits the contract, the agent produces a scoped code patch, a receipt card appears with the delta banner); PostToolUse hook keeps the graph live AND journals session intent to `.contracts/journal/<session-id>.jsonl`; cross-level propagation detects rollup staleness on every cache rebuild and offers pin-aware reconcile from amber-flagged upstream nodes
**Depends on**: Phase 4, Phase 5, Phase 7
**Requirements**: AGENT-01, AGENT-02, AGENT-03, AGENT-04, CHRY-01, CHRY-02, CHRY-03, MCP-02, PROP-01, PROP-02, PROP-03, PROP-04
**Success Criteria** (what must be TRUE):
  1. User types an intent in the chat panel; the app runs `claude` CLI scoped to the currently-zoomed node's contract and its neighbors (not a whole-repo grep); the agent's streaming output appears in the chat panel in real time
  2. After the agent run, a receipt card appears in the inspector showing non-zero token counts, tool call count, estimated cost, and nodes touched — parsed defensively from the session JSONL with a fixture-based mock fallback preventing a blank card; JSONL parser ships as an isolated module with unit tests against a captured real session
  3. The cherrypick flow produces a side-by-side diff modal with a persistent orientation header (`NodeName — intent — N tool calls`) above contract / code / preview diffs; user approves both in one action; sidecar and source file are written atomically via a single Rust IPC command
  4. Receipt cards persist per node and are retrievable from the inspector's receipt history tab; user pins two receipts side-by-side and the comparison view leads with a full-width delta banner (e.g., `−82% tokens  −83% tool calls  −85% wall time`) in 28px+ high-contrast type
  5. PostToolUse hook fires after Claude Code file writes, **appends one JSONL line per edit to `.contracts/journal/<session-id>.jsonl`** (extracting `intent` from `$CLAUDE_TRANSCRIPT_PATH`'s latest user prompt; falls back to a thin tool_use summary for headless `-p` invocations), re-derives affected L3/L4 contracts via `update_contract`, and updates drift state through the same per-node Tokio Mutex as Phase 7's watcher — hook and watcher coexist without racing
  6. Target-node ring glow appears on selection *before* any edit (CHRY-01 bridge from graph to inspector focus) and persists through the agent run — verified visually on camera
  7. **Section hashes are computed by the canonical Rust section parser** (`src-tauri/src/sidecar/section_parser.rs` — fenced-code-aware, order-stable, rejects duplicate headings) and persisted on every `write_derived_contract` / `update_contract`. Both committed fixtures (`11111111-…` API L3, `22222222-…` UI L4) round-trip stable `section_hashes` despite different section ordering; MCP sidecar computes hashes via IPC to Rust rather than maintaining a parallel TypeScript implementation
  8. **Rollup detection** recomputes `rollup_state` for every L1/L2/L3 node on cache rebuild (startup, file-watcher events, post-write). A node with unchanged code but a child whose cited `section_hash` changed flips to `rollup_state: stale` and pulses amber within 2 seconds of the child write. A node with `rollup_inputs: []` shows gray (`untracked`). Precedence red > amber > gray holds when multiple conditions apply on the same node
  9. **Pin-aware reconcile panel.** Unpinned amber node → three actions: **Draft propagation for review** (force-shows proposed diff before commit — framing per red-team Jin & Chen 2026 / Stengg 2025), **Accept as-is** (L1 requires one-line justification persisted to journal), **Edit manually**. Pinned amber node → three different actions: **Review children's changes** (read-only diff of drifted sections since last `rollup_generation`), **Unpin and reconcile** (two-step), **Accept as-is, keep pin** (with justification). `SKIPPED-PINNED` is unreachable from both paths — the branching fires before any writer is called

  1. **Cmd+P substrate search.** User opens Cmd+P, types a free-form rationale question (e.g., `"why email confirmation"`), and gets ranked substrate hits across `verbatim_quote` + `summary` + `applies_when` (FTS5 + DeepSeek listwise rerank, reusing the same primitive Phase 11 retrieval already uses). New `Substrate` filter chip joins existing `Contracts` / `Code` chips. Selecting a hit opens the existing `SourceArchaeologyModal` in <2s end-to-end (keystroke → readable verbatim quote rendered).
  2. **Refine with chained version.** `SourceArchaeologyModal` exposes a `Refine` button. Click → modal expands; rule `text` and `applies_when` become editable; reason field required. Save creates a new `substrate_nodes` row with `prev_version_uuid` pointing to the previous row's UUID; the previous row's `invalid_at` is set to now with `invalidated_reason = 'refined: <user reason>'`. Modal gains a `History` tab that walks the chain via `prev_version_uuid` and renders each version with a diff against the prior.
  3. **Audit table.** New `substrate_edits` table captures every refine / delete / restore: `{edit_id, rule_uuid, prev_version_uuid, new_version_uuid, actor (e.g., 'human:yangg40@g.ucla.edu'), edited_at, before_text, after_text, reason, kind ∈ {refine, delete, restore}}`. Audit row is written atomically with the substrate-node version write (single Rust IPC, single SQLite transaction). Audit table survives reset-demo.sh (preserved across the demo reset path).
  4. **Delete with mandatory reason + impact preview.** `Delete this rule` button (red). Confirm dialog requires (a) reason picker as a radio: `Hallucinated · Obsolete · Wrong scope · Duplicate · Other`; (b) free-text amplification (required when `Other`, optional otherwise); (c) auto-loaded **impact preview** showing atoms that cite the rule (count + list, queried via `atom_substrate_refs` or equivalent join) and recent agent prompts in the past 7 days that included the rule (count, queried from receipts referencing it). Confirm sets `invalid_at = now`, `invalidated_reason = '<picker>: <text>'`, `invalidated_by = 'human:<email>'`. Tombstone preserved indefinitely.
  5. **Substrate Health sidebar + Restore.** New `Substrate Health` sidebar surface (accessible from a sidebar icon) lists tombstoned rules with `kind`, `invalidated_reason`, `invalidated_at`, and a `Restore` action. Restore sets `invalid_at = NULL` on the most recent tombstoned version, writes a `restore` audit row, and surfaces a toast confirming the rule is active again. No time limit on restore (unlike many soft-delete systems).
  6. **Phase 12 supersession compatibility.** Phase 12 supersession queries (fact-level invalidation, intent-level supersession walker) audited under chained versions: queries that filter `WHERE invalid_at IS NULL` continue to return the current-version row; the rerank in `retrieval/rerank.rs` reads only current rows. Smoke: refine an existing fixture rule (`con-stripe-customer-archive-2026-02-22` → narrower text), run a fresh Beat 4 agent task — the **new** wording (not the old) appears in the agent's substrate hits. Phase 12 intent-drift cascade still flips correctly when an L0 priority shift fires through the chain.
  7. **Beat 3 fixture → real path.** T's narrowing of `con-settings-no-modal-interrupts-2025-Q4` in Beat 3 (currently a staged fixture per `13-10a` substrate seed) recoded against the real refine path. Audit row visible in modal History tab post-edit. Reproducible 5× from `reset-demo.sh` reset state. Stage time unchanged (refine completes in <2s on the demo machine).

**Planning notes**:
  - **Demo placement (recommended):** B + C from the scoping conversation. Option B promotes the Beat 3 fixture into a real interaction; Option C reserves Cmd+P substrate search and the delete path as Q&A weapons. Option A (visible Refine/Delete buttons in Beat 3 narration without clicking) is wasted on script viewers who won't notice. The demo timing budget is unchanged — the refine action takes the same on-screen seconds as the current fixture animation.
  - **Schema migration coordination.** `prev_version_uuid` adds a self-referential FK on `substrate_nodes`. A new `substrate_edits` table joins on `rule_uuid`. Folds into a new numbered migration (next available — verify against Phase 14 in flight). Existing substrate rows backfill `prev_version_uuid = NULL` (they're the head of their chain).
  - **Current-version view.** A new `current_substrate_view` SQL view filters to `WHERE invalid_at IS NULL` and exposes the same column set as `substrate_nodes`. Phase 12 queries that do current-truth filtering should be audited and migrated to read from this view (or keep their current `WHERE invalid_at IS NULL` predicate — both work; the view exists for clarity, not speed).
  - **FTS5 reindex.** New / refined rows must reindex into the existing substrate FTS5 virtual table. Tombstoned rows must be removed from the index so search results don't include deleted rules. Reuse the existing FTS5 trigger pattern from Phase 11.
  - **Why mandatory reason on delete.** The reason itself becomes substrate-quality evidence — *"we caught a hallucination, here's how, here's what we did."* It also makes the demo answer to *"what stops the substrate from filling with hallucinated rules?"* tangible: open rule → Delete → reason picker → impact preview → confirm. Eight seconds. The reason picker is the punchline.
  - **Impact preview query.** For each rule, two counts: (1) atoms citing the rule via `atom_substrate_refs` join (or whatever Phase 11 named the link table); (2) receipts in the past 7 days where this `rule_uuid` appears in the prompt context. Both queries should return in <500ms on the demo SQLite snapshot.
  - **Edit-conflict handling.** If two users (or the user + the distiller) attempt to refine the same rule concurrently, last-writer-wins on the chain — the second writer's `prev_version_uuid` will point to the first writer's new row, not the original. This is acceptable for v1; v2 could add optimistic-lock checks via a `version_generation` field if it ever matters in practice.
  - **Restore semantics.** Restore brings back the most recent tombstoned version (`invalid_at` cleared), not arbitrary historical versions. Restoring a rule whose chain has subsequent refines should be impossible — once you restore, the chain head is the restored version. v2 can add "branch" semantics if needed.
