<objective>
Scaffold the Tauri v2 + React + TypeScript project with all macOS native chrome, Monaco CSP, and build tooling wired correctly at scaffold time. This is pure foundation — every later phase assumes these decisions are already made.

Purpose: Retrofitting Monaco CSP, vibrancy, or CSP `blob:` later is expensive and error-prone. Per research (RESEARCH.md pitfalls 1-4), three scaffold-time items are zero-cost if applied now and multi-day debugging sessions if missed: (1) no `#[tokio::main]`, (2) Monaco + `vite-plugin-monaco-editor` + `blob:` CSP, (3) `macOSPrivateApi: true` with `transparent: true` for vibrancy to survive production builds.

Output: A `cargo tauri dev`-runnable Tauri v2 project with a translucent window, native traffic lights, SF Pro font rendering, vibrant sidebar, Monaco worker support, and shadcn/ui initialized. No business logic yet — just the shell.
</objective>

    ```json
      "minHeight": 700,
      "decorations": true,

      "titleBarStyle": "Overlay",
      "trafficLightPosition": { "x": 19, "y": 24 },
      "hiddenTitle": true
    }
    ```
    And set `app.macOSPrivateApi: true` (top-level inside `app`) — required for transparency + vibrancy to survive `cargo tauri build` per Pitfall 3 / GitHub #13415.

    Add CSP under `app.security`:
    ```json
    "csp": {
      "default-src": ["'self'"],
      "script-src": ["'self'", "blob:"],
      "style-src": ["'self'", "'unsafe-inline'"],
      "img-src": ["'self'", "data:", "asset:", "https://asset.localhost"],
      "connect-src": ["'self'", "ipc:", "https://ipc.localhost"]
    }
    ```
    `blob:` in `script-src` is what lets Monaco's `new Worker(URL.createObjectURL(...))` succeed inside WKWebView (Pitfall 2).

    Edit `contract-ide/src-tauri/capabilities/default.json` so the `permissions` array includes:
    `"shell:allow-execute"`, `"shell:allow-spawn"` — the claude-CLI subprocess test in Plan 04 requires these. SQL permissions will be added in Plan 02.

    Edit `contract-ide/src-tauri/src/lib.rs` (the Tauri v2 scaffold puts the builder there, not `main.rs`). Replace the `.run()` block so the builder registers `tauri_plugin_shell::init()` AND applies vibrancy in the setup hook:
    ```rust
    // NEVER add #[tokio::main] — Tauri owns the runtime (RESEARCH.md Pitfall 1).
    use tauri::Manager;
    use window_vibrancy::{apply_vibrancy, NSVisualEffectMaterial};

    #[cfg_attr(mobile, tauri::mobile_entry_point)]
    pub fn run() {
        tauri::Builder::default()
            .plugin(tauri_plugin_shell::init())
            .setup(|app| {
                let window = app.get_webview_window("main").unwrap();
                #[cfg(target_os = "macos")]
                apply_vibrancy(&window, NSVisualEffectMaterial::Sidebar, None, None)
                    .expect("apply_vibrancy failed — macOSPrivateApi must be true in tauri.conf.json");
                Ok(())
            })
            .run(tauri::generate_context!())
            .expect("error while running tauri application");
    }
    ```
    Confirm `src-tauri/src/main.rs` contains ONLY the scaffolded `fn main() { contract_ide_lib::run(); }` — no `#[tokio::main]`, no tokio import. Add a comment at the top: `// NEVER add #[tokio::main] here — Tauri owns the runtime.`

    Edit `contract-ide/vite.config.ts` to register `vite-plugin-monaco-editor`:
    ```ts
    import { defineConfig } from 'vite';
    import react from '@vitejs/plugin-react';
    import monacoEditor from 'vite-plugin-monaco-editor';

    export default defineConfig({
      plugins: [
        react(),
        (monacoEditor as any).default({
          languageWorkers: ['editorWorkerService', 'typescript', 'json', 'css', 'html'],
        }),
      ],
      clearScreen: false,
      server: { port: 1420, strictPort: true },
    });
    ```
    The `(monacoEditor as any).default` cast is required because the plugin ships CJS and the default export is nested under `.default` when imported via ESM — verified in RESEARCH.md Standard Stack; using a bare call silently no-ops the plugin.

    Keep any existing Tauri-specific Vite options the scaffold wrote (e.g., `envPrefix`, HMR config); only add to the `plugins` array and replace `server` if the scaffold did not set one.

    Do not add vibrancy to non-main windows and do not try to scope vibrancy to a subregion — whole-window vibrancy + CSS background-color on non-sidebar panels is the approach (RESEARCH.md Open Question 4). The CSS override lands in Plan 03.
