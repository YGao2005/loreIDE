**delete_substrate_rule** (signature exactly):

```rust
#[tauri::command]
pub async fn delete_substrate_rule(
    app: tauri::AppHandle,
    uuid: String,
    reason_kind: String,    // "Hallucinated" | "Obsolete" | "Wrong scope" | "Duplicate" | "Other"
    reason_text: String,    // free-text; required when reason_kind == "Other"
    actor: String,          // "human:<email>"
) -> Result<(), String>
```

Implementation:
1. Validate `reason_kind` is one of the 5 allowed values; return Err("invalid reason_kind") otherwise.
2. Validate that `reason_kind == "Other"` implies `!reason_text.trim().is_empty()`; return Err("free-text required when reason is Other") otherwise.
3. Read the old row text (single SELECT WHERE uuid=?1 AND invalid_at IS NULL); if None, return Err("rule {uuid} not found or already tombstoned").
4. `now = chrono::Utc::now().to_rfc3339()`. Compose `invalidated_reason = format!("{reason_kind}: {reason_text}")`.
5. `let mut tx = pool.begin().await?;`
6. UPDATE substrate_nodes SET invalid_at=?now, invalidated_reason=?invalidated_reason, invalidated_by=?actor WHERE uuid=?uuid AND invalid_at IS NULL.
7. INSERT into substrate_edits: edit_id=new_v4, rule_uuid=uuid (the affected row, no new chain row created on delete), prev_version_uuid=NULL, new_version_uuid=NULL, actor=actor, edited_at=now, before_text=old_text, after_text=NULL, reason=invalidated_reason, kind='delete'.
8. tx.commit().
9. The FTS UPDATE trigger fires automatically — removes the row from substrate_nodes_fts and does NOT re-INSERT (because invalid_at is now NOT NULL).
10. Return Ok(()).

Implementation per RESEARCH §Pattern 6:

```sql
-- Atoms citing the rule via anchored_uuids JSON array
SELECT n.uuid, n.name, n.kind, n.level
FROM nodes n, substrate_nodes s, json_each(s.anchored_uuids) je
WHERE s.uuid = ?1 AND je.value = n.uuid
LIMIT 50

-- Total atom count (separate query — could COUNT(*) the above without LIMIT)
SELECT COUNT(*) AS atom_count
FROM substrate_nodes s, json_each(s.anchored_uuids) je
WHERE s.uuid = ?1
```

```sql
-- Recent prompts (past 7 days) that included the rule
-- Reads receipts.substrate_rules_json column landed in plan 15-01
SELECT id, created_at, COALESCE(prompt_summary, '') AS prompt_excerpt
FROM receipts
WHERE created_at > datetime('now', '-7 days')
  AND substrate_rules_json IS NOT NULL
  AND substrate_rules_json LIKE '%' || ?1 || '%'
ORDER BY created_at DESC
LIMIT 50

-- Recent prompt count
SELECT COUNT(*) AS recent_prompt_count
FROM receipts
WHERE created_at > datetime('now', '-7 days')
  AND substrate_rules_json IS NOT NULL
  AND substrate_rules_json LIKE '%' || ?1 || '%'
```

NOTE: The LIKE '%uuid%' approach is a coarse string match because substrate_rules_json is a JSON array string. This is acceptable because UUIDs are unique enough that false positives are statistically zero. If receipts table doesn't have a `prompt_summary` column, replace with `''` literal; the `prompt_excerpt` field is best-effort and may be empty in v1. Read receipts schema in `db/migrations.rs` to confirm available columns; pick the closest existing column (e.g., `intent_text` or `intent` if present) for the excerpt.
