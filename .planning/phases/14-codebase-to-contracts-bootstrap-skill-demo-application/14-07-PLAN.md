<objective>
Replace the "No contracts yet — open a repo" empty state with an in-IDE Bootstrap CTA that runs the codebase-to-contracts skill end-to-end and animates new nodes onto the graph as the pipeline produces them — turning the bootstrap from a Q&A-only inset into the load-bearing demo affordance.

Today the skill is invocable only via `/codebase-to-contracts` inside a Claude Code session. The output is a `.contracts/` directory the user must then point the IDE at as a separate step. The new flow folds the bootstrap into the IDE itself: open a Next.js repo, see "Bootstrap with Skill," click, watch nodes appear on the graph in real time as Stage 1 discovers, Stage 2 derives, Stage 4 aligns, and Stage 5b atomically emits.

Architecture is build-on-existing-rails:
- Phase 7 file watcher (`drift/watcher.rs`) already triggers `refresh_nodes` on `.contracts/` mutations.
- Phase 11 distiller (`distiller/pipeline.rs:323`) already emits `substrate:nodes-added` Tauri events that the canvas animates against (HarvestPanel pattern).
- This plan reuses both patterns. Bootstrap progress events follow the same naming + payload shape as `substrate:nodes-added` so the canvas animation hooks generalize.
- The skill scripts already exist (preflight, discover, derive-frontmatter, derive-body, align-jsx, synthesize-flows, emit). We add structured stderr JSONL progress emission to each — minimal surgery — and a Rust orchestrator that spawns them in sequence and pipes events.

Output: a clean "open repo → no contracts → bootstrap CTA → real-time graph population → done" flow demoable in a single take, with a recordable progress UX that reads as deliberate engineering, not an opaque agent black box.

This is the WAVE 5 plan — depends on Plan 14-05 for atomic emit. Tasks 1-3 (Rust orchestrator + script JSONL emission) can land before 14-04/05 against current Stage 1-2 output; Tasks 4-7 (UI + end-to-end) require the full pipeline so the demo's "graph fully populated" state is real, not partial.
</objective>

    ```javascript
    export function emitProgress(event) {
      // Each event is one JSONL line on stderr — never stdout, which is
      // reserved for skill output piping. Rust parses these line-by-line.
      process.stderr.write(JSON.stringify({
        ts: new Date().toISOString(),
        ...event,
      }) + '\n');
    }
    ```

    Each script emits a fixed set of event shapes:

    - `{event: 'stage', stage: 1|2|3|4|5|'5a'|'5b', status: 'started'|'completed', node_count?: number}` — emitted at top + bottom of each stage's main function
    - `{event: 'node', stage: ..., uuid, kind, level, file, route?}` — emitted as each node lands in `.staging/` (frontmatter.json or body.md or final .md)
    - `{event: 'flow', stage: '5a', uuid, slug, member_count}` — emitted per flow contract
    - `{event: 'cost-estimate', estimate_usd: number, node_count: number}` — preflight only
    - `{event: 'error', stage: ..., diagnostic: {file?, uuid?, message}}` — on any stage failure

    Instrumentation points (one-line additions per script — minimal surgery):

    - `preflight.mjs`: emit `cost-estimate` after the file count + heuristic estimate; emit `stage 0 started/completed` around the shape check.
    - `discover.mjs`: emit `stage 1 started` at top; emit `node` per appended candidate (for L4 atoms one-per-PascalCase-decl, for everything else after heuristic classification); emit `stage 1 completed` with node_count.
    - `derive-frontmatter.mjs`: emit `stage 2 started`; emit `node` per `.staging/<uuid>.frontmatter.json` write (existing OR new — re-runs need to render too); emit `stage 2 completed`.

    Do NOT change stdout output of any script — that path is the existing CLI UX. The progress emission is purely additive.
