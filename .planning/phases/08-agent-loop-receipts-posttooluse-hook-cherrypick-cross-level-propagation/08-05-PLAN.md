       ```rust
       use std::path::{Path, PathBuf};
       use serde::{Deserialize, Serialize};

       #[derive(Debug, Clone, Serialize, Deserialize)]
       pub struct FilePatch {
           pub file: String,        // path relative to repo root
           pub new_content: String,
       }

       #[tauri::command]
       pub async fn apply_cherrypick(
           app: tauri::AppHandle,
           uuid: String,
           contract_body: String,
           file_patches: Vec<FilePatch>,
       ) -> Result<(), String>
       ```

       Implementation order — load-bearing per Pitfall 6:
       a. Resolve repo path from `RepoState` (Plan 07-02 lineage: fetch via `app.state::<RepoState>()`, scope `std::sync::MutexGuard` BEFORE any `.await`).
       b. Acquire DriftLocks: `let locks = app.state::<crate::drift::state::DriftLocks>(); let _guard = locks.for_uuid(&uuid).lock().await;` — same lock as compute_and_emit + compute_rollup_and_emit so the watcher and rollup engine wait until the cherrypick completes (RESEARCH.md decision: ONE lock per UUID, no namespacing).
       c. Validate every `FilePatch.file` is INSIDE the repo (call existing canonicalize+containment guard from Plan 04-01 `read_file_content` — STATE.md: target.starts_with(repo) assertion). Reject any path that escapes the repo with `Err("path escapes repo: ...")`.
       d. **Compute all temp paths first.** For each FilePatch: `tmp_path = file.with_extension("tmp.cherrypick")` (or use `<file>.<uuid>.tmp` to avoid collisions). For sidecar: `sidecar_tmp = .contracts/<uuid>.md.tmp.cherrypick`.
       e. **Phase 1 — write all temp files (NO renames yet).** For each FilePatch, `std::fs::write(&tmp_path, &new_content)` + fsync (use `std::fs::OpenOptions::new().write(true).create(true).truncate(true).open(...)?.sync_all()?` for explicit fsync; if File::sync_all fails on the dev macOS, drop fsync but document the deviation). For the sidecar: build new content using the existing `crate::sidecar::frontmatter::write_sidecar(fm, body) -> String` helper (B4 — actual name is `write_sidecar`, NOT `build_sidecar_content`; verified at sidecar/frontmatter.rs:80). Pass through `compute_section_hashes(contract_body)` from 08-01 IF AVAILABLE (graceful degradation: if 08-01 lands later, ship the cherrypick without section_hashes recompute and 08-01 backfills via lazy migration on next write); recompute `contract_hash` via existing `hash_text` helper. Write to `sidecar_tmp` + fsync.
       f. **Phase 2 — rename source temps FIRST.** For each FilePatch in order: `std::fs::rename(&tmp_path, &final_file_path)`. If ANY rename fails, return `Err(format!("partial-cherrypick: source file {file} rename failed: {e}"))` — DO NOT roll back already-renamed source files (no atomic 2PC on POSIX). State at this point: at least one source file updated, sidecar still pointing at OLD contract_hash → Phase 7 SourceWatcher fires `compute_and_emit` on the next FSEvents tick → user sees red pulse → can retry. **Failure is observable.**
       g. **Phase 3 — rename sidecar LAST.** `std::fs::rename(&sidecar_tmp, &sidecar_final_path)`. If this fails, source files are updated but sidecar didn't move → Phase 7 watcher still fires (source file mtime changed → recomputes code_hash → drift detected against old contract_hash). Observable.
       h. On success, return `Ok(())`. The frontend's invocation-success path closes the modal and emits a small toast.

       NOTE on filesystem boundaries: `std::fs::rename` is atomic on the SAME filesystem. If source files and `.contracts/` are on different mounts (rare in this project), rename across filesystems falls back to copy + unlink which is NOT atomic. Detect this case (catch `errno::EXDEV`) and return `Err("cross-filesystem rename — cherrypick requires repo root and .contracts/ on the same mount")`. Document in SUMMARY.

       NOTE on race with PostToolUse hook (08-03): the hook is read-only on SQLite and writes only to `.contracts/journal/`. apply_cherrypick writes to `.contracts/<uuid>.md` and source files. No race — the hook never touches the sidecar body or source files; it only journals.
