<objective>
Land MCP-02 + PROP-03: a PostToolUse hook that does TWO things on every agent file write — (1) journals the edit to `.contracts/journal/<session_id>.jsonl` (Pass 1, intent record), and (2) for each affected UUID spawns a fire-and-forget background `claude -p` subprocess that calls the `update_contract` MCP tool to rederive the contract body from observed code (Pass 2, fresh-eyes verification per the three-pass model). Plus a Rust IPC for reading the journal entries (consumed by 08-06's reconcile panel).

**The product thesis** (load-bearing): the contract is the verification surface for the agent's *implicit* decisions. Designers, PMs, and skim-reading engineers verify the agent's work by reading contracts, NOT by reading bulk code. If contracts go stale (drift surfaces but contract body lags), the verification surface lies. **Auto-rederive is therefore load-bearing, not optional.**

**Three-pass verification model** (architecturally agreed; Phase 8 ships passes 1 + 2):
- **Pass 1 — subjective truth (intent):** working agent journals `{intent, file, ...}` per file write. The hook script does this synchronously before exit.
- **Pass 2 — objective truth (observed behavior):** fresh agent reads code, rederives contract body. This plan ships this pass via the per-UUID `claude -p` spawn. Owns the contract's `## Examples` and `## Implicit Decisions` sections. User-authored `## Intent` and `## Role` are preserved (NOT overwritten).
- **Pass 3 — alignment truth:** verifier checks intent vs observed against substrate rules. Phase 11+ (NOT Phase 8). Beat 3's "Verify against intent" panel IS Pass 3.

**Why a SEPARATE `claude -p` (fresh agent) and not the active session calling MCP:**
- Fresh-eyes pass — the active session is biased by "I just wrote this" and systematically misses implicit decisions (the very thing contracts must surface)
- Single-session-recursion avoidance — calling `update_contract` from inside the active agent loop creates weird recursion
- Subscription auth carries to subprocess; no API key needed (Phase 6 derivation pivot pattern)
- Mirrors Phase 6 derivation: LLM derives contract body from code

**Race / single-writer rule** (Pitfall 2 reframing):
- "Single-writer" means "serialize all writes through `DriftLocks::for_uuid(uuid)` Tokio mutex" — NOT "only one writer exists"
- The hook spawns `claude -p` which calls `update_contract` MCP tool, which writes the .md file via existing `write_contract` path
- Phase 7 watcher fires on the .md file write (FSEvents) → recomputes drift
- All writers (hook→claude-p→update_contract AND active-session→update_contract via future Phase 11 Delegate button) coexist because mutex coordination handles the race
- No new write path introduced by this plan — the hook is a *client* of the existing `update_contract` path

**Latency expectation** (document explicitly, surface in SUMMARY):
- 5-file agent run → ~5 background `claude -p` spawns, ~10-30s each, parallel
- Visible behavior: brief red pulse (drift detected on raw .ts/.tsx write) → fresh (Pass 2 rederive completes, contract matches code, drift clears) over ~30s
- This is a FEATURE: visibly shows the verification loop working, better than instant magic-update

Output: Hook script (with both journal write AND per-UUID claude-p rederive spawn) + Claude settings.json + Rust journal-read IPC + TS IPC wrapper + unit tests including a test that asserts the rederive spawn fires (mock or counts subprocess invocations). The rederive write path is bash → claude CLI → MCP `update_contract` (existing — Phase 5 shipped); Rust only READS journal files in this plan.
