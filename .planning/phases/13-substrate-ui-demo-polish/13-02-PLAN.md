**Step 1 — Rust command `get_sidebar_tree` in `commands/sidebar.rs`.**

The "area" concept derives from the directory structure under `.contracts/`. Each top-level subdirectory of `.contracts/` is an area (e.g. `.contracts/auth/`, `.contracts/billing/`, `.contracts/account/`). Flows are contracts with `kind = 'flow'` (Phase 9 FLOW-01 ships this kind). The IPC returns:

```rust
#[derive(Debug, Serialize, Clone)]
pub struct SidebarFlow {
    pub uuid: String,
    pub name: String,
    pub kind: String,        // always 'flow'
    pub member_uuids: Vec<String>,  // for fast badge aggregation in frontend
}

#[derive(Debug, Serialize, Clone)]
pub struct SidebarArea {
    pub area: String,                    // e.g. 'auth' (relative dir under .contracts)
    pub member_uuids: Vec<String>,       // ALL contract uuids whose file_path is under this area
    pub flows: Vec<SidebarFlow>,
}

#[tauri::command]
pub async fn get_sidebar_tree(
    db_instances: State<'_, DbInstances>,
) -> Result<Vec<SidebarArea>, String> {
    // 1. Query all nodes with their file_path
    // 2. Group by top-level dir under .contracts/ (use path-component split, take first non-empty)
    // 3. For each area: collect uuids; subselect kind='flow' nodes; for each flow read its `members` array from the contract body or sidecar (the FLOW-01 schema stores members in YAML frontmatter — fetch via existing parse_sidecar machinery, or via a new column if Phase 9 added one)
    // 4. Sort areas alphabetically; sort flows alphabetically
}
```

**Sourcing the `members` field:** Phase 9 ships FLOW-01 — `kind: flow` contracts have `members: [uuid, ...]` in YAML frontmatter. This plan executes BEFORE Phase 9 ships members, so until then `members` is unknown. **Defensive impl:** if the node's parsed frontmatter has `members`, use it; otherwise fall back to empty array. The sidebar still renders flow names; the chain layout in plan 13-06 reads members directly when populated.

Use `crate::sidecar::frontmatter::parse_sidecar` against the sidecar path (existing Phase 2 pattern) inside a `tauri::async_runtime::spawn_blocking` to avoid blocking the async runtime on disk I/O.

**Defensive area extraction:** For a `file_path` like `.contracts/auth/login.md`, area = `auth`. For `.contracts/account.md` (top-level file), area = `_root`. For `.contracts/`, area = `_root`. Top-level area name conflicts are unlikely at hackathon scale; document inline.

**Step 4 — Zustand store at `src/store/sidebar.ts`:**

```typescript
import { create } from 'zustand';
import type { SidebarArea } from '@/ipc/sidebar';

interface SidebarState {
  tree: SidebarArea[];
  expandedAreas: Set<string>;
  selectedFlowUuid: string | null;
  setTree: (tree: SidebarArea[]) => void;
  toggleArea: (area: string) => void;
  setSelectedFlow: (uuid: string | null) => void;
}

export const useSidebarStore = create<SidebarState>((set) => ({
  tree: [],
  expandedAreas: new Set(),
  selectedFlowUuid: null,
  setTree: (tree) => set({ tree }),
  toggleArea: (area) =>
    set((s) => {
      const next = new Set(s.expandedAreas);
      if (next.has(area)) next.delete(area);
      else next.add(area);
      return { expandedAreas: next };
    }),
  setSelectedFlow: (uuid) => set({ selectedFlowUuid: uuid }),
}));
```

**Step 1 — `SidebarAreaItem.tsx` — single area row.**

Renders:
- Click-to-toggle row showing area name + chevron
- Three small badges (drift count, rollup-stale count, intent-drifted count) when nonzero — color-coded matching CVA variants from plan 13-01 (red dot for drift, amber for rollup-stale, orange for intent-drifted)
- When expanded, lists child flows below

Uses three Zustand selectors with custom equality for performance:

```typescript
import { useDriftStore } from '@/store/drift';
import { useRollupStore } from '@/store/rollup';
import { useSubstrateStore } from '@/store/substrate';
import { useSidebarStore } from '@/store/sidebar';

interface Props { area: SidebarArea; }

export function SidebarAreaItem({ area }: Props) {
  const expanded = useSidebarStore((s) => s.expandedAreas.has(area.area));
  const toggle = useSidebarStore((s) => s.toggleArea);

  // Per-area badge counts — recompute when relevant store changes; member_uuids is stable
  const driftCount = useDriftStore((s) =>
    area.member_uuids.filter((u) => s.driftedUuids.has(u)).length
  );
  const rollupStaleCount = useRollupStore((s) =>
    area.member_uuids.filter((u) => s.rollupStaleUuids.has(u)).length
  );
  const intentDriftedCount = useSubstrateStore((s) =>
    area.member_uuids.filter((u) => s.nodeStates.get(u) === 'intent_drifted').length
  );

  return (
    <div data-area={area.area}>
      <button
        onClick={() => toggle(area.area)}
        className="flex w-full items-center px-2 py-1 hover:bg-muted/50 rounded"
      >
        <ChevronIcon expanded={expanded} />
        <span className="text-sm">{area.area === '_root' ? 'Root' : area.area}</span>
        <span className="ml-auto flex gap-1">
          {driftCount > 0 && <Badge variant="drift">{driftCount}</Badge>}
          {rollupStaleCount > 0 && <Badge variant="rollup">{rollupStaleCount}</Badge>}
          {intentDriftedCount > 0 && <Badge variant="intent">{intentDriftedCount}</Badge>}
        </span>
      </button>
      {expanded && area.flows.length > 0 && (
        <div className="ml-4 border-l border-border/50">
          {area.flows.map((flow) => <SidebarFlowItem key={flow.uuid} flow={flow} />)}
        </div>
      )}
    </div>
  );
}
```

`Badge` here is a small inline component — 4-letter pill, 18px height, color-coded. DO NOT use shadcn Badge if it adds dep weight; just inline a `<span class="inline-flex h-4 min-w-[16px] items-center justify-center rounded-full text-[10px] ...">` with variant-specific bg/text colors. Match exact hex from contractNodeStyles.ts to keep visual consistency: drift = red-500, rollup = amber-500, intent = orange-600.

`SidebarFlowItem` (defined in same file or a sibling) is a click-to-select row for the flow. Calls `useSidebarStore.setSelectedFlow(flow.uuid)` AND `useGraphStore.pushParent(flow.uuid)` (or whichever action navigates the canvas — check Phase 3 03-02 patterns). Plan 13-06 will refine the canvas response; for now this hook just changes the selected flow.

**Step 2 — `SidebarTree.tsx` — top-level tree.**

```typescript
import { useEffect } from 'react';
import { useSidebarStore } from '@/store/sidebar';
import { getSidebarTree } from '@/ipc/sidebar';
import { SidebarAreaItem } from './SidebarAreaItem';

export function SidebarTree() {
  const tree = useSidebarStore((s) => s.tree);
  const setTree = useSidebarStore((s) => s.setTree);

  useEffect(() => {
    getSidebarTree().then(setTree).catch((e) => console.error('[sidebar] tree load:', e));
    // Re-fetch on `nodes:refreshed` event (existing Phase 2 watcher event) — keeps badges live
    // Use the same pattern as drift/rollup subscriptions in AppShell.
  }, [setTree]);

  if (tree.length === 0) {
    return <div className="px-3 py-2 text-xs text-muted-foreground">No contracts loaded</div>;
  }

  return (
    <div className="flex flex-col">
      {tree.map((area) => <SidebarAreaItem key={area.area} area={area} />)}
    </div>
  );
}
```

Subscribe to existing watcher event so the tree refreshes when contracts are edited. Look at `AppShell.tsx` for how `drift:changed` / `rollup:changed` are subscribed and follow that pattern with a `nodes:refreshed` (Phase 2) or `repo:opened` event.
