**Helper:**

```rust
use std::path::Path;
use tauri::AppHandle;
use tauri::Manager;

/// Read every node's (uuid, list-of-code-range-files) from SQLite and
/// re-register the source-file watcher. Called from open_repo AND
/// refresh_nodes so a brand-new node's code_ranges start being watched
/// within the sidecar-refresh window (Pitfall 7).
pub async fn refresh_source_watcher_from_db(app: &AppHandle, repo_path: &Path) {
    let instances = app.state::<tauri_plugin_sql::DbInstances>();
    let map = instances.0.read().await;
    let Some(db) = map.get("sqlite:contract-ide.db") else { return };
    let pool = match db {
        tauri_plugin_sql::DbPool::Sqlite(p) => p,
        _ => return,
    };

    let rows: Vec<(String, String)> = sqlx::query_as(
        "SELECT uuid, code_ranges FROM nodes WHERE code_ranges IS NOT NULL"
    )
    .fetch_all(pool)
    .await
    .unwrap_or_default();

    let mut pairs: Vec<(String, Vec<String>)> = Vec::new();
    for (uuid, ranges_json) in rows {
        let ranges: Vec<crate::sidecar::frontmatter::CodeRange> =
            serde_json::from_str(&ranges_json).unwrap_or_default();
        let files: Vec<String> = ranges.into_iter().map(|r| r.file).collect();
        if !files.is_empty() {
            pairs.push((uuid, files));
        }
    }

    let watcher = app.state::<crate::drift::watcher::SourceWatcher>();
    if let Err(e) = watcher.refresh(app.clone(), repo_path, &pairs) {
        eprintln!("[drift] SourceWatcher::refresh failed: {e}");
    }
}
```

Important details:
- Releases the `DbInstances` read lock (via `drop(map)` or letting `map` fall out of scope) BEFORE calling `watcher.refresh`. The watcher's internal `notify::recommended_watcher` setup is blocking; holding the DB read lock through it would serialize scans.
- `ranges_json` deserialize failures fall through to `unwrap_or_default()` — a malformed sidecar row doesn't poison the whole refresh. Existing `get_nodes` handler already uses this pattern.

**Tauri command `acknowledge_drift`:**

```rust
#[tauri::command]
pub async fn acknowledge_drift(
    app: AppHandle,
    uuid: String,
) -> Result<(), String> {
    // Acquire per-UUID mutex BEFORE any DB work so a concurrent watcher
    // event for the same uuid can't re-flag drifted between our UPDATE
    // and emit.
    let locks = app.state::<crate::drift::state::DriftLocks>();
    let mutex = locks.for_uuid(&uuid);
    let _guard = mutex.lock().await;

    let instances = app.state::<tauri_plugin_sql::DbInstances>();
    let map = instances.0.read().await;
    let db = map.get("sqlite:contract-ide.db").ok_or("DB not loaded")?;
    let pool = match db {
        tauri_plugin_sql::DbPool::Sqlite(p) => p,
        _ => return Err("only sqlite supported".into()),
    };

    let now = chrono::Utc::now().to_rfc3339();

    // UPDATE only — the node must already have a drift_state row to
    // acknowledge. If no row exists, Ok(()) silently (idempotent UX).
    sqlx::query(
        "UPDATE drift_state SET reconciled_at = ?2 WHERE node_uuid = ?1"
    )
    .bind(&uuid)
    .bind(&now)
    .execute(pool)
    .await
    .map_err(|e| e.to_string())?;

    // Release DB lock before emitting (defensive — event handlers on the
    // frontend may turn around and invoke another Rust command).
    drop(map);

    // Emit drift:changed { drifted: false } to clear the red pulse.
    // Use serde_json::json! for the one-off payload — camelCase matches
    // the engine.rs DriftChanged struct's serialized shape.
    use tauri::Emitter;
    let _ = app.emit("drift:changed", serde_json::json!({
        "uuid": uuid,
        "drifted": false,
        "currentCodeHash": null,
        "baselineCodeHash": null,
    }));

    Ok(())
}
```

Key correctness notes:
- Signature returns `Result<(), String>`: frontend needs Ok/Err to toast on failure; unlike `probe_route` (which collapses to bool), acknowledge has meaningful failure modes (DB lock, missing row → though we treat missing as Ok).
- Payload fields match `engine::DriftChanged` camelCase shape so the React listener can use one event-handler signature for both paths.
- No `human_pinned` check needed — acknowledging is always allowed regardless of pin state (the user is explicitly overriding drift flag).

- The call happens BEFORE returning the scan_result to the frontend — the frontend's next action is probably to call `get_nodes`, which should arrive at a point where the watcher is already registered (otherwise a user who edits a file between scan-complete and watcher-register would miss that edit).

- DO NOT propagate errors from the watcher refresh back to the frontend. Log via `eprintln!` inside the helper (already done in Task 1). A failure to register the watcher is a degraded mode (no drift detection) but the repo open should still succeed — the user can click "Reopen repo" to retry.

**`refresh_nodes`:** This command currently updates a single sidecar row. Per RESEARCH Pitfall 7, we MUST refresh the watcher so a new node's code_ranges start being watched. BUT: `refresh_nodes` may be called many times per second during a bulk derive run (each `write_derived_contract` triggers the Phase 2 fs watcher → `refresh_nodes`). A full rebuild on every call is cheap at hackathon scale (< 200 files), but we should still check the repo_state first: if `repo_state` is `None`, skip entirely.

// Phase 7: re-register the source-file watcher so any newly-derived
