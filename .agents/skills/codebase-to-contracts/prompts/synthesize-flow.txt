# Stage 5a — Verify Flow Members Ordering

You are verifying the `members:` ordering of a Contract IDE format-v5 flow
contract. A static AST walk has already collected a candidate ordered list
of UUIDs by traversing the trigger's import graph + recursing one level into
each callee. Your job is one focused task: confirm or correct the
INVOCATION-ORDER of those members for this flow's purpose.

You output a single JSON object that conforms to `schemas/flow.json`:

  {
    "members": [<trigger-uuid>, <participant-uuid>, ..., <terminal-uuid>],
    "notes": "Numbered invocation walkthrough — becomes ## Notes body."
  }

----------------------------------------------------------------------
SECTION A — MISSION

A flow contract's `members:` array MUST list participants in actual
runtime invocation order. The first element is always the trigger
(typically a UI L3 page or API L3 endpoint); subsequent elements are
the libs / data accesses / external API calls fired in the order they
execute when the trigger fires.

The static walker that produced the candidate ordering walked the AST in
preorder by line number — which is usually right but can fail on:

  - **Dynamic dispatch.** `const fn = handlers[op]; await fn(...)` — the
    walker can't see which handler is invoked. ADD the actual handler if
    you can identify it from the trigger source.
  - **Conditional branches.** `if (X) { await foo() } else { await bar() }`
    — both paths got walked; confirm only the dominant path's order.
  - **Dead-code imports.** `import { unused } from './lib'` — the
    walker may have proposed a member that's imported but not actually
    invoked. REMOVE dead members.
  - **Helper-call ordering inside callees.** The walker is one-level
    deep; if `beginAccountDeletion()` calls `db.user.update` BEFORE
    `stripe.customers.update`, that ordering must be preserved in the
    flow's members list (db.user.update comes before stripe.customers.update).
  - **Promise.all parallel calls.** Walker emits sequential ordering;
    if two members fire in parallel via `Promise.all([...])`, the order
    in the flow is the order they appear in the array literal.

You will receive in the user prompt:
  - The flow trigger's UUID + file path + kind (UI L3 or API L3).
  - The trigger source code (first 3000 chars).
  - The candidate ORDERED members list (Stage 5a static walker output),
    each annotated with `<uuid> — <file> (<kind>)`.

You output:
  - `members`: the corrected ordered UUID array. Same shape as the
    proposal, with members reordered / removed / added as needed.
  - `notes`: the ## Notes body for the flow contract — a numbered walk
    explaining the invocation chain in prose. Match the rhythm of the
    Phase 9 flow exemplar (see SECTION C).

----------------------------------------------------------------------
SECTION B — RULES

1. **First element is always the trigger UUID.** Even if you reorder
   everything else, the trigger UUID stays at index 0.
2. **Use only UUIDs from the candidate set.** Do NOT invent UUIDs. If
   the walker missed a participant, you may not add it (the static
   scan + AST walk is the source of truth on which contracts EXIST in
   the codebase). The only valid additions are UUIDs FROM the
   candidate list that you re-add to the ordered output.
3. **Preserve unique UUIDs.** Each UUID appears at most once in the
   members list (Stripe.customers.update called twice in the chain
   still emits ONE member entry; the flow contract is a participant
   list, not an event log).
4. **Never reorder past the trigger.** members[0] = trigger always.
5. **Notes prose density target: ≥30 chars, ≥3 numbered steps when
   the flow has ≥3 participants.** Match the exemplar's voice.

----------------------------------------------------------------------
SECTION C — EXEMPLAR (Phase 9 flow-delete-account.md ## Notes body)

  Member ordering is invocation order:
  1. Account Settings page (trigger UI) — user clicks Delete Account
  2. POST /api/account/delete (API endpoint) — soft-delete kicks off
  3. beginAccountDeletion lib — orchestrates the full 5-rule chain
  4. db.user.update — sets User.deletedAt (soft-delete marker)
  5. stripe.customers.update — archives customer (NOT customers.del)
  6. mailchimp.suppress — sets list status to unsubscribed
  7. sendDeletionConfirmationEmail — email-link confirmation (24h expiry)

The 7-member chain above is the prose-density target. Each numbered step
names the participant (without UUID), states what it does in 5-10 words,
and adds one specificity callout (the soft-delete marker, the archive-
NOT-delete distinction, the 24h expiry). YOUR notes body should match
this rhythm for whatever flow you're synthesizing.

----------------------------------------------------------------------
SECTION D — COMMON CORRECTIONS

If the candidate list reads:

  1. <trigger-uuid> — src/app/account/page.tsx (UI)
  2. <auth-uuid> — src/lib/auth.ts (lib)
  3. <delete-uuid> — src/lib/account.ts (lib)
  4. <stripe-uuid> — src/lib/stripe.ts (external)

But the trigger calls auth FIRST as a guard (and bails on unauth),
then delete, then stripe — that ordering is correct, you just
confirm it.

If the candidate list reads:

  1. <trigger-uuid> — src/app/dashboard/page.tsx (UI)
  2. <stripe-uuid> — src/lib/stripe.ts (external)
  3. <auth-uuid> — src/lib/auth.ts (lib)

But the trigger source clearly shows `await getSession(req)` BEFORE
`await stripe.customers.list()`, swap members[1] and members[2] —
the walker was tripped by line-number preorder when the imports were
declared in a different order than usage.

Output the JSON. No prose outside the JSON object.
